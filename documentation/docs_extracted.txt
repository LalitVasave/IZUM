Security Considerations
Password stored as bcrypt hash with cost factor 12 — never plain text, never MD5/SHA1.
Email check endpoint is rate-limited (20 req/min per IP) to prevent email enumeration.
college_id and phone number are hashed (SHA-256) before storage — not searchable in plain form.
JWT uses RS256 (asymmetric) — private key signs, public key verifies — private key never leaves server.
Registration endpoint rate-limited to 5 attempts/hour/IP to prevent mass account creation.
For production v1, email verification is optional but recommended. The verified flag in the users table is pre-built for this — flip it to required when email relay is configured.

Auth Page
Purpose & Scope
The Login page authenticates returning users, issues a short-lived access token and a long-lived refresh token, and routes them to their role-appropriate dashboard. The token strategy is designed for production security: access tokens live only in JS memory (never localStorage), refresh tokens live in httpOnly cookies (XSS-proof).

axios.interceptors.response.use(null, async (error) => {
  if (error.response?.status === 401) {
    await axios.post("/auth/refresh");   // cookie sent automatically
    return axios(error.config);          // retry original request
  }
  return Promise.reject(error);

Dead Reckoning Loop — TypeScript
// Runs every 100ms via requestAnimationFrame
const deadReckon = () => {
  const { lat, lng, heading, speed } = useVehicleStore.getState();
  const elapsed = 0.1; // 100ms in seconds
  const dist    = (speed / 3.6) * elapsed; // speed in km/h → m/s
  const newLat  = lat + (dist / 111320) * Math.cos(toRad(heading));
  const newLng  = lng + (dist / 111320) * Math.sin(toRad(heading));
  markerRef.current?.setLatLng([newLat, newLng]);
  requestAnimationFrame(deadReckon);
};
// On real WS ping — divergence check
const onPing = (real: Position) => {
  const dist = haversine(interpolated, real);
  if (dist <= 5) smoothCorrect(real);    // lerp over 500ms
  else           snapTo(real);           // snap + expand circle
};

Performance Considerations
requestAnimationFrame runs at display refresh rate (60fps) but dead reckoning only updates position every 100ms — throttled with a timestamp check to avoid over-computation.
Leaflet marker setLatLng() is called only when position delta exceeds 0.5m — prevents redundant DOM updates during very slow movement.
Redis pub/sub broadcast is fire-and-forget — Fastify WS handler does not await acknowledgement from frontend, preventing head-of-line blocking across connected clients.
Stop + route data is fetched once on load and cached in Zustand — never re-fetched during the session unless the user switches routes.

Uncertainty Bar Calculation
// Python — FastAPI ML service
base_width     = 2.0  # minutes — minimum range at perfect signal
signal_factor  = 1 + (signal_tier * 0.5)        # tier 0→1x, tier 3→2.5x
staleness_factor = 1 + (ping_gap_seconds / 90)  # grows linearly
confidence_width = base_width * signal_factor * staleness_factor
eta_min = max(0, base_eta_minutes - confidence_width / 2)
eta_max = base_eta_minutes + confidence_width / 2
# uncertainty_bar_pct — for frontend visual
uncertainty_bar_pct = min(100, int((confidence_width / 15) * 100))

1. System Architecture Overview
IZUM is a resilience-first campus bus tracking platform built on a TypeScript + Python full-stack. The architecture is designed around three core principles: GPS dead reckoning for continuous position interpolation, an ML-powered ETA model using 8 signal-aware features, and an honest status system that communicates network degradation without alarming users.

3.1 Dead Reckoning Algorithm
When no WS ping arrives, the frontend projects position every 100ms using the last known speed and heading. This ensures the marker always moves — even on degraded signal.
Variable
Formula / Logic
Distance per frame (m)
dist = (speed_kmh / 3.6) × 0.1  →  converts km/h to m/s, multiplies by 100ms
New latitude
newLat = lat + (dist / 111320) × cos(toRad(heading))
New longitude
newLng = lng + (dist / 111320) × sin(toRad(heading))
Divergence check
haversine(interpolated, real_ping) — if ≤ 5m: lerp correction over 500ms; if > 5m: snap + expand circle
Throttle guard
timestamp diff check — actual position update only if > 100ms since last; rAF runs at 60fps
DOM optimisation
setLatLng() skipped if position delta < 0.5m — prevents redundant DOM updates at near-zero speed

5. Safety Feature Architecture
5.1 Silent SOS (Page 7)
Three rapid taps on a near-invisible shield icon (opacity 20%) within 800ms triggers a 10-second cancel window, then silently dispatches to campus security. No visible alarm. No red text. Haptic feedback only on mobile.
Design Decision
Rationale
Shield icon at opacity-20
Invisible to bystanders — student can trigger without being seen
3 taps in < 800ms required
Prevents accidental triggers from normal map interaction
10-second cancel window
Breathing room for accidental triggers — cancel bar drains visibly at top of screen
No SOS text on screen
Bystander-proof — no visual tells. Only haptic vibration on the student's phone
Student ID from JWT, not body
Prevents spoofed SOS — server extracts student_id from token sub claim
GPS + bus position fallback
If geolocation API is denied, uses Redis bus:{id}:position as location
student_id hashed in DB
SHA-256 hash — SOS events are pseudonymous; not linked to personal records by default
Rate limit: 3 SOS/student/hr
Prevents spam while preserving genuine emergency capacity
Resend email within 2s
Campus security alert sent before SOS row is even committed to DB
5.2 Virtual Escort (Page 8)
A student shares a token-based URL (expires in 2 hours) with a parent or warden. The escort viewer polls every 15 seconds — no login required, no app install. Shows bus position, route progress, and arrival status.
5.3 Late Night Mode (Page 9)
Activates automatically after 21:00. Overlays a deep dark theme on the map and adds a 10-minute driver check-in countdown ring. A missed check-in fires an immediate security alert via Resend. Route deviation beyond 100m highlights expected vs actual path.
